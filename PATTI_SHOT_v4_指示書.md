## 2. 何を作るか（プロダクト定義）

FireShot Capture 相当のフルページスクリーンショットツール。Windows用の無料アプリ。

**利用者の体験：**
1. プロダクトページから `PATTI_SHOT.exe` をダウンロード
2. ダブルクリックすると、専用のブラウザウィンドウが開く
3. 普通にネットを見る（ログインが必要なページは普通にログインすればよい。次回以降も保持される）
4. 撮りたいページで、**画面右下に浮いているピンクの「PATTI SHOT」ボタン**をクリック
5. スクロールしないと見えない一番下まで丸ごと撮影され、PNG / PDF が保存される
6. 保存フォルダが自動で開く

**これだけ。** 利用者が触るファイルは `PATTI_SHOT.exe` ただ1つ。設定ファイルもフォルダ構成も見せない。

## 3. 技術構成

- 言語：Python
- ブラウザ制御：Playwright
- **利用者がインストール済みのChromeを起動する**（`channel="chrome"`）。専用のユーザープロファイルを永続保存し、ログイン状態を保持する
  - Chrome未インストール環境では、初回のみ Chromium を自動取得するフォールバックを用意する（利用者に何も操作させない）
- UI：`add_init_script` で**全ページにフローティングUIを注入**する
  - 右下に丸いピンクのPATTI SHOTボタン
  - ボタン長押し or 歯車アイコンで設定パネル（出力形式 PNG/PDF、解像度倍率 1x/2x/3x）
  - 進行状況（`撮影中 4/18` 等）と完了通知も注入UI内に表示
  - **注入UIは撮影対象に写り込ませないこと**（撮影直前に非表示化し、完了後に復帰）
- 配布：PyInstaller によるワンファイル `.exe`
- 保存先：`ダウンロード\PATTI SHOT\` に固定（初回に自動作成）。保存後にそのフォルダをエクスプローラーで開く

## 4. 撮影パイプライン仕様（v3から継承・ここが品質の本丸）

### STEP 1: 前処理

1. **スクロール主体の特定**：`document.scrollingElement` を基本としつつ、全要素を走査して `scrollHeight - clientHeight > 200` かつ面積最大の要素を検出。それが実際のスクロール主体ならそちらを対象にする
2. **遅延読み込みの発火**：スクロール主体を最下部まで往復スクロール。打ち切り上限 100,000px。各ステップで `requestAnimationFrame` 2回＋80ms待機
3. **描画の強制**：`content-visibility: auto` を一時的に `visible` へ、`loading="lazy"` を `eager` へ、`<img>` の `decode()` を待つ
4. 一番上へ戻す

### STEP 2: 実コンテンツ高さの実測（既知バグの根本原因・最重要）

**`scrollHeight` を信用しない。** 業務系サイト（例：J-PlatPat）は中身より遥かに大きい高さを申告してくるため、これを鵜呑みにすると巨大な空白が発生する。

```
実測高さ = 全可視要素（display:none / visibility:hidden / 面積0 を除く）の
          getBoundingClientRect().bottom + scrollY の最大値
撮影高さ = min(scrollHeight, 実測高さ + 余白100px)
```

さらに撮影後の画像に対して**末尾の均一色帯を自動トリミング**する（実測漏れの保険）。

### STEP 3: 撮影

- 原則：`deviceScaleFactor` を設定したうえでフルページ撮影
- **超ロングページ対策**：一発撮りには縦方向の描画限界があるため、限界を超える場合は**座標指定（clip）による区間分割撮影 → 結合**に自動で切り替える。スクロール合成ではなくレンダリング座標指定のため、固定ヘッダーの重複が発生しない
- **固定要素の無害化**：分割時のみ、`position: fixed` / `sticky` の要素を一時退避。撮影後は元のstyle属性を必ず復元する
- **空白自動検出＋自動再撮影**：撮影結果を後述の判定にかけ、空白帯が検出されたら**利用者に何も聞かず**、解像度を下げる／分割数を増やす等の条件変更で自動リトライ（最大3回）。それでも失敗した場合のみ、日本語で理由を表示する

### STEP 4: 出力

**PNG**
- 巨大画像は単一のバッファに依存せず、タイルから逐次エンコードして書き出す
- 失敗時のフォールバック：解像度を段階的に落として再構成 → それでも不可なら分割ファイル出力し、その旨をUIに明記

**PDF**
- 基本は「見た目そのままの縦長1ページ」
- PDFの1ページ最大寸法は 14,400pt（200インチ）という規格上の上限があるため、超える場合は**複数ページへ自動分割**（横幅統一、継ぎ目でテキスト行が切れないよう分割位置を微調整）
- 文字の可読性を優先し、過度なJPEG圧縮をしない

**ファイル名**：`PATTISHOT_ドメイン名_YYYYMMDD_HHMMSS.png` / `.pdf`

## 5. 自動更新（AUTOPILOT方式の発展形）

- 起動時に GitHub Releases の最新バージョンを確認
- 新しい版があれば、注入UI内に「新しいバージョンがあります／更新する」ボタンを表示
- クリックすると新しいexeを取得し、更新用の小さなバッチ（ASCII限定/CRLF/BOMなし/chcpなし）が旧exeを置き換えて再起動する
- **利用者にzip解凍もフォルダ操作もさせない**
- 通信失敗時は黙って通常起動する（更新確認の失敗でアプリが止まらないこと）

## 6. 自動検証ハーネス（最重要・ボスをテスターにしないための中核）

本アプリはPlaywrightベースなので、**Code自身がスクリプトから全機能を検証できる**。人間の実機確認に依存する実装にしないこと。

`test/` 配下に検証スクリプトを作成し、以下を**機械的に判定**する。

| 判定 | 合格基準 |
|---|---|
| 空白帯検出 | 画像を高さ100pxの帯に分割。単一色が99.5%以上を占める帯が**連続300px以上**続いたら FAIL（末尾の意図的トリム後の余白は除く） |
| 高さ整合 | 出力画像の高さ ÷ 解像度倍率 が、STEP2の実測高さ ±3% に収まる |
| 重複検出 | 行のハッシュを取り、ビューポート高さ間隔で同一の長い連続行が繰り返されていたら FAIL（固定ヘッダー重複） |
| 欠落検出 | DOM上のテキストからランダムに10個のキーワードを抽出し、その要素の座標が画像内で空白でないことを確認 |
| 注入UI写り込み | 出力画像の右下領域にPATTI SHOTボタンが写っていないこと |
| 解像度 | 画像の横幅 = ビューポート幅 × 指定倍率 と一致 |
| 後始末 | 撮影後、ページのstyleが撮影前と一致（固定要素の復元漏れ検出） |
| エラー | 全操作を通してJavaScriptエラー0件、Python例外0件 |

### テスト対象ページ（12件以上・実在ページを選定し記録すること）

1. **J-PlatPat 商標検索結果ページ（必須。検索操作もスクリプトで自動化する）** ← 既知の失敗ケース
2. 縦20,000px以上の極端に長いページ
3. 遅延読み込み画像が多いページ
4. 固定ヘッダー＋追従バナーがあるページ
5. 内側スクロールコンテナを持つ業務系ページ
6. 無限スクロール系ページ（打ち切りが正しく効くか）
7. 短い1画面ページ
8. 横幅が広いページ（横スクロールあり）
9. 日本語フォント中心のページ
10. テーブルが多いページ
11. iframeを含むページ
12. ダークテーマのページ（空白検出が誤作動しないか）

結果は `test/REPORT.md` に自動生成し、失敗画像は `test/artifacts/` に保存すること。

> **検証時の条件（ウィンドウサイズ、解像度倍率、OS）を報告に明記すること。** 条件を再現できないテストはPASSとみなさない。
